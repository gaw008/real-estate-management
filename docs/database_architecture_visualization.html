📊 数据库表结构

🏠 Properties (房产表)
  • id VARCHAR(20) PK
  • name VARCHAR(100)
  • city VARCHAR(50)
  • state VARCHAR(30)
  • street_address TEXT
  • bedrooms INT
  • bathrooms DECIMAL(3,1)
  • property_size INT
  • occupancy INT
  • is_active BOOLEAN
👥 Owners_Master (业主表)
  • owner_id VARCHAR(20) PK
  • name VARCHAR(200)
  • phone VARCHAR(50)
  • email VARCHAR(500)
  • preferences_strategy TEXT
  • residence VARCHAR(100)
  • language VARCHAR(50)
  • total_properties INT
🔗 Property_Owners (关系表)
  • property_id VARCHAR(20) PK,FK
  • owner_id VARCHAR(20) PK,FK
  • is_primary BOOLEAN
💰 Finance (财务表)
  • property_id VARCHAR(20) PK,FK
  • cleaning_fee DECIMAL(10,2)
  • management_fee_rate DECIMAL(5,2)
  • management_fee_type ENUM('Net','Gross')
  • contract_signed_date DATE
  • listing_date DATE
  • first_booking_date DATE
🔑 表设计要点：
  • Properties表为核心主表，包含房产的所有基本信息
  • Owners_Master独立存储业主信息，避免数据冗余
  • Property_Owners关系表支持一房多主的复杂关系
  • Finance表专门存储财务相关信息，便于财务分析

📋 关系说明

🔄 一对多关系 (1:N)
  • Owners_Master → Property_Owners 一个业主可拥有多个房产
  • Properties → Property_Owners 一个房产可有多个业主
🔗 一对一关系 (1:1)
  • Properties → Finance 每个房产对应一条财务记录
-- 外键约束示例 ALTER TABLE property_owners ADD CONSTRAINT fk_property_owners_property FOREIGN KEY (property_id) REFERENCES properties(id) ON DELETE CASCADE; ALTER TABLE property_owners ADD CONSTRAINT fk_property_owners_owner FOREIGN KEY (owner_id) REFERENCES owners_master(owner_id) ON DELETE CASCADE; ALTER TABLE finance ADD CONSTRAINT fk_finance_property FOREIGN KEY (property_id) REFERENCES properties(id) ON DELETE CASCADE;

⚡ 性能优化策略

🔍 索引优化
  • 复合索引：idx_location_features (state, city, bedrooms)
  • 覆盖索引：idx_listing_cover 避免回表
  • 全文索引：FULLTEXT (name, street_address)
  • 范围索引：idx_date_range 时间查询
🎯 分区策略
  • 地理分区：HASH(CRC32(state)) 10个分区
  • 时间分区：RANGE(YEAR(contract_date))
  • 分区裁剪：自动排除无关分区
  • 并行查询：多分区并行处理
💾 缓存策略
  • Redis缓存：热点数据 TTL 30分钟
  • MySQL缓存：查询缓存 256MB
  • 应用缓存：统计数据 TTL 1小时
  • CDN缓存：静态资源加速
🔧 配置优化
  • InnoDB缓冲池：1GB
  • 连接池：最大200连接
  • 日志文件：256MB
  • 表缓存：2000个表
